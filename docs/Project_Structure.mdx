## File Naming Conventions

### **Services:**
- `{domain}_service.py` (e.g., `account_service.py`)
- Class: `{Domain}Service` (e.g., `AccountService`)

### **Repositories:**
- `{domain}_repository.py` (e.g., `user_repository.py`)
- Class: `{Domain}Repository` (e.g., `UserRepository`)

### **Models:**
- `{domain}.py` (e.g., `user.py`, `post.py`)
- Class: `{Entity}` (e.g., `User`, `Post`)

### **API:**
- `views.py`, `serializers.py`, `urls.py` per domain
- Versioned in `api/v1/`, `api/v2/`, etc.

## Import Conventions

While PEP 8 generally recommends absolute imports, Django projects follow a specific convention to ensure apps are reusable and self-contained.

https://docs.djangoproject.com/en/5.2/intro/reusable-apps/

- **Intra-App Imports (within the same app):** Use **explicit relative imports**. This prevents tying an app to a specific project name, making it modular and reusable.

  ```python
  # Correct: In accounts/views.py
  from .models import User
  from .serializers import UserSerializer
  ```

  ```python
  # Incorrect:
  from glade.accounts.models import User
  ```

- **Inter-App Imports (between different apps):** Use **absolute imports**. This makes dependencies between apps clear and explicit.

  ```python
  # Correct: In posts/views.py, importing a model from the accounts app
  from accounts.models import User
  ```
